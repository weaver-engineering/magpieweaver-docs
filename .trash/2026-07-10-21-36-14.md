# Local Dev Environment Setup

This document supports [task TBD-01](../tasks/TBD-01/task-TBD-01.md) and describes how to stand up the
[Development Architecture](../architecture.md#development-architecture) on a clean machine, and how to confirm
everything still works when resuming development after a break.

The Development Architecture has four moving parts, all running locally:
1. Browser SPA (React/TS UI)
2. TS Service & Git binary (local compute)
3. LM Studio (local LLM)
4. Local FS (per-user workspaces & Git repos) and local RAM (session cache)

> **Note:** Several details below (exact tool versions, package manager, port numbers, env var names) are marked
> `<CONFIRM>` because they depend on decisions not yet made elsewhere in the docs (e.g. ADR-001 Tech Stack Solution).
> Fill these in once the code repository exists and replace this note.

## 2. Clone the repositories

Magpie Weaver is split across two repositories (see [Repositories](../architecture.md#repositories)):

```bash
mkdir -p ~/dev/magpie-weaver
cd ~/dev/magpie-weaver

git clone https://github.com/simonemmott/magpieweaver.git code
git clone https://github.com/simonemmott/magpieweaver-docs.git docs
```

The code repo is cloned into the agent's workspace so it can read the architecture and task documentation
alongside the code it's working on, per [Development Approach](../architecture.md#development-approach).

---

## 3. Install dependencies

```bash
cd ~/dev/magpie-weaver/code
pnpm install
```

This should install dependencies for both the React/TS UI and the TS Service.

## 5. Set up LM Studio

1. Install and open LM Studio.
2. Download a local model. `<CONFIRM which model — this is tied to the open "LLM model TBD" item in architecture.md>`
3. Start LM Studio's local server (Developer tab → Start Server), and confirm it's listening on the port referenced

## 6. Create the local workspace storage directory

```bash
mkdir -p ~/dev/magpie-weaver/workspaces
```

This stands in for the EFS-backed per-user workspace storage used in test/production
(see [Storage Tier Isolation](../ADRs/ADR-004-storage-tier-isolation.md)).

## 7. Start the TS Service

```bash
cd ~/dev/magpie-weaver/code
pnpm dev
```

Confirm it starts without errors and is listening on the expected port.

## 8. Start the UI

```bash
cd ~/dev/magpie-weaver/code
pnpm dev:ui
```

Open the app in your browser at `<CONFIRM URL, e.g. http://localhost:5173>`.

---

## 9. First-run smoke test

Confirm the full stack is wired together correctly:

- [ ] UI loads in the browser without console errors.
- [ ] UI can reach the TS Service (check network tab for a successful call to a basic endpoint, e.g. health check).
- [ ] TS Service can reach LM Studio (trigger any action that calls the LLM and confirm a response comes back).
- [ ] TS Service can read/write to the local workspace directory (create a test workspace and confirm files appear
      under `~/dev/magpie-weaver/workspaces`).
- [ ] Git binary is callable from the TS Service (trigger any action that commits to a per-user Git repo and confirm
      the commit appears with `git log`).

If all five pass, the environment is ready for task work.

---

## 10. Resuming after a break

When picking development back up after time away, before starting new task work:

1. Pull latest on both `code` and `docs` repos.
2. Re-run step 3 (`install`) in case dependencies changed.
3. Confirm LM Studio still has the correct model loaded and its server is running (step 5).
4. Re-run the step 9 smoke test in full — don't assume the environment is still in the state you left it.
5. Check Linear for any tasks that moved phase while you were away, and re-read the relevant `task-<REF>.md` /
   `task-<REF>-spec.md` before resuming.
